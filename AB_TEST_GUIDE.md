# 🧪 A/B 테스트 실행 가이드

## 📋 테스트 개요

이 가이드는 대기열 시스템의 효과를 측정하기 위한 A/B 테스트를 단계별로 안내합니다.

**목표:** 대기열 시스템이 사용자 경험과 시스템 안정성에 미치는 영향을 정량적으로 측정

---

## 🎯 테스트 목표

### **비교할 두 시나리오**

#### **Group A (대기열 사용)**
- 사용자가 SmartQueueGate를 통해 번호표 발급
- 순서대로 입장 허용
- sessionToken으로 보안 강화
- **장점:** 서버 과부하 방지, 사용자 순서 보장
- **단점:** 대기 시간, 이탈 가능성

#### **Group B (대기열 없음)**
- 바로 신청 페이지로 이동
- Rate Limiting 완화
- sessionToken만으로 보안
- **장점:** 빠른 신청, 간단한 프로세스
- **단점:** 동시 접속 시 서버 과부하 가능

### **측정할 KPI**

| 지표 | 정의 | 중요도 |
|------|------|--------|
| 전환율 | 페이지 뷰 중 신청 완료 비율 | ⭐⭐⭐⭐⭐ |
| 성공률 | 신청 시작 중 성공 비율 | ⭐⭐⭐⭐⭐ |
| 이탈률 | 페이지 뷰 후 이탈 비율 | ⭐⭐⭐⭐ |
| 평균 소요시간 | 신청 완료까지 평균 시간 | ⭐⭐⭐ |
| 서버 에러율 | 서버 오류 발생 비율 | ⭐⭐⭐⭐ |

## 🚀 Phase 1: 준비 단계

### **Step 1.1: A/B 테스트 설정**

1. 관리자 페이지 접속: `https://project-id.web.app/admin`
2. 학교 설정 선택
3. "6. A/B 테스트 설정" 섹션에서:
   ```
   ✅ A/B 테스트 활성화: 체크
   ✅ Group A 할당 비율: 50%
   ✅ 테스트 시작 시간: (현재 시간)
   ✅ 테스트 종료 시간: (7일 후)
   ```
4. "설정 저장" 클릭

### **Step 1.2: 모니터링 대시보드 확인**

1. 같은 페이지에서 다음 패널 확인:
   - 실시간 시스템 통계 (SystemStatsPanel)
   - 신청 트렌드 분석 (AnalyticsPanel)
   - A/B 테스트 결과 (ABTestResultsPanel)

2. 각 패널의 데이터 로딩 확인:
   - 활성 Rate Limits 수
   - 대기 AlimTalk 수
   - 최근 활동 내역

### **Step 1.3: AlimTalk 테스트**

1. 테스트 사용자 생성
2. 수동 신청 진행
3. AlimTalk 수신 확인
4. 템플릿 파라미터 검증

## 🧪 Phase 2: 파일럿 테스트

### **Step 2.1: 테스트 계정 생성**

```javascript
// 테스트 사용자 계정 (100명 분)
// 1. Google 계정 100개 준비 (또는 Firebase Auth REST API)
for (let i = 1; i <= 100; i++) {
    // 자동화 스크립트로 계정 생성 가능
}
```

### **Step 2.2: 테스트 시나리오**

**시나리오 1: 정상 트래픽**
- 100명이 10분 동안 접속
- 예상: 50명 Group A, 50명 Group B

**시나리오 2: 스파이크 트래픽**
- 100명이 동시에 접속 (30초 내)
- 예상: Group A는 대기열 분산, Group B는 과부하 가능

**시나리오 3: 정원 마감 직전**
- 확정 5자리 남은 상태에서 20명 접속
- 예상: Group B는 빠르게 마감, Group A는 순차 대기

### **Step 2.3: 데이터 수집**

**실시간 모니터링:**
```bash
# Firebase Functions 로그 모니터링
firebase functions:log --only registerRegistration
firebase functions:log --only processAlimTalkQueue
firebase functions:log --only autoAdvanceQueue
```

**Firestore 데이터 확인:**
```bash
# Firebase Console → Firestore → Data
registrationMetrics: 100건 확인
abTestMetrics: A/B 그룹별 메트릭 확인
```

### **Step 2.4: 초기 결과 분석**

**예상 데이터:**
```

- 페이지 뷰: 50명

- 신청 완료: 45명 (실패 3명)
- 전환율: 90%
- 성공률: 94%

Group B (대기열 없음):
- 페이지 뷰: 50명
- 신청 시작: 50명
- 신청 완료: 40명 (실패 10명)
- 전환율: 80%
- 성공률: 80%
```

## 🎯 최종 결정

### **권장 사항: Group A (대기열 시스템) 채택**

**이유:**
1. 전환율과 성공률이 모두 높음
2. 서버 안정성이 크게 개선됨
3. 사용자 경험이 더 좋음
4. 스케일 아웃 시 확장 가능

### **개선 권장사항:**

**1. 대기열 경험 개선:**
- 예상 대기 시간 더 정확히 표시
- 대기열 진행 상태 실시간 알림
- 대기 시간 동안 흥미로운 콘텐츠 제공

**2. Group B 문제 해결:**
- 서버 과부하 방지를 위해 Rate Limiting 유지
- 에러 메시지 더 명확하게 제공
- 실패 시 재시도 안내

**3. 하이브리드 접근:**
- 정원 충분할 때: 대기열 사용
- 정원 여유가 많을 때: 대기열 스킵 가능

## 🔧 문제 해결 가이드

### **문제 1: A/B 테스트 데이터 부족**

**해결:**
- 테스트 기간 연장 (7일 → 14일)
- 파일럿 테스트 인원 증원
- 리마인더/알림 발송

### **문제 2: 양 그룹 차이 미미**

**해결:**
- 테스트 설정 변경 (50:50 → 70:30)
- 대상 그룹 세분화 (신규/기존)
- 테스트 기간 연장

### **문제 3: 기술적 이슈**

**해결:**
- Cloud Functions 로그 확인
- Firebase Functions 배포 확인
- Firestore 인덱스 재생성
